> 本文档是根据 experiment_plan.md 和 SKILL （# Elfiee MCP Tools）列出的缺失的tools，以及在缺失的情况下，dogfooding 过程中和产品实际使用过程中分别需要人工执行的替代工作。

| 缺失内容                                                      | Dogfooding 阶段的替代方案                                                           | 产品实际使用中的替代方案                                          |
| --------------------------------------------------------- | ---------------------------------------------------------------------------- | ----------------------------------------------------- |
| **Task Type（任务分组字段）**                                     | PM 在创建任务时人工保证 task type 命名一致；实验分析阶段由 PM 人工判断哪些任务属于同一类型、区分 first / second use | 用户自行记忆或通过全文搜索判断“是否做过类似任务”；产品侧无法提供基于任务类型的学习、推荐或统计      |
| **implement 关系（Task → Code）**                             | Dev 在开发过程中尽量自觉建立关联；PM 在归档与复盘时人工判断哪些代码属于该 task                                | 用户通过 commit message、文件 diff 或记忆判断代码归属；系统无法自动完成精确归档与复用 |
| **Clarification（需求/规则澄清）**                                | PM 在任务结束后回放对话记录，人工标注哪些轮次属于澄清，用于统计 Clarification Count                        | 无可行替代方案；用户不会主动标注澄清，系统也无法判断澄清行为，相关学习指标缺失               |
| **test_result / FPY（一次通过）**                               | Dev 口头或在文档中说明测试是否一次通过；PM 在复盘时人工判定 FPY                                        | 无可行替代方案；系统无法判断效率提升，用户也不会结构化记录测试结果                     |
| **Summary（阶段性总结与采纳）**                                     | PM 通过对比任务前后描述、最终产出，主观判断 AI 是否理解并被采纳                                          | 用户不会额外生成或确认总结；系统只能学习结果，无法学习“理解是否被采纳”                  |
| **task.commit（任务完成语义）**                                   | PM 人工判断 task 何时算完成，并手动对齐对应的 Git commit，用于粗略计算 TTE                            | 用户自行记忆 task 与 commit 的对应关系；系统无法提供清晰的任务生命周期            |
| **Task 状态机（Pending / InProgress / Committed / Archived）** | PM 在实验记录中人工维护 task 状态，用于分析时间节点                                               | 用户自行管理任务状态；系统无法基于状态提供分析、提醒或学习                         |
| **PM 验收测试（Acceptance Result）**                            | PM 在实验记录或复盘文档中人工写明是否通过验收                                                     | 用户通常不会记录验收结果；系统无法区分“做完了”和“做好了”                        |
| **Archive（稳定复用入口）**                                       | PM 人工维护“哪些任务值得复用”的列表或文档；Second-use 依赖 PM 记忆                                  | 用户只能全文搜索或凭印象找旧任务；系统无法主动学习或推荐历史经验                      |
| **History Reference 的结构化统计**                              | PM 在复盘时人工判断是否参考了历史任务 / skill                                                 | 系统无法判断是否发生学习行为；产品无法证明“第二次更好”来自历史复用                    |
