# 1. 概述

AI带来的效率悖论体现在技术发展的各个层面中：
1. 应用层面：AI的加入产生了更多冗余且模糊的需求，也产生了更多复杂且低效的实现。原来60分可以达成的供需关系（找到历史文档），现在需要做到80分（找到文档->总结关键结论->分角色总结->后续行动），而供需双方都付出了100分的成本（多元异构数据->权限合规->复杂策略），甚至无法满足最初60分的验收标准。
2. 模型层面：通用性更强，专用难度增加（工程、对齐、验证成本增加）。
3. 数据层面：数据使用量提升，提升数据质量的难度加大。
4. 基建层面：算力密度提升，算力、电力等调度成本提升。

以往的思考在于，如何继续从更适配的基础设施，更多的数据标注和清晰工作，更高效的模型架构等方面，bottom-up的解决问题。在当前野性投资，市场成交量却不高的情况下，需要TOP-DOWN的思考问题，从应用层面拉齐供需的角度去解决问题，避免过量的投入。

# 2. 痛点（What）

从应用的效率悖论出发，可以看到并不是“AI不够聪明”，而是在需求方与供给方之间，缺少一个可以同时承载意图、实现、验证和逻辑链条的中间媒介，把供需双方在应用层面拉齐。

1. 大部分需求以 PRD、会议纪要、聊天记录等形式存在，这些文本在 AI 时代更容易被快速改写和总结，容易不可控的不断扩大。开发者（人或 Agent）最终还是要“猜”：哪些是硬性约束？哪些是背景信息？哪些是验收标准？

2. Git 能很好地记录代码的版本变化，但无法记录理由（reasoning）。在AI加入的多环节协作中，这种“决策失忆”会被放大。新加入的 Agent/开发者很难判断哪些逻辑可以重构，场景方也难以追溯某个结果是来自哪一次假设、哪一次妥协。

3. 传统模式下，通过人工测试或少量脚本，确认现在这个版本能满足当前场景。进入 AI 时代后，场景和模型都在持续变化，但很少有系统会把验收标准和理由沉淀为可以自动回放的资产，把“场景描述 – 测试用例 – 实际行为 – 评价结果”绑定在一起。每次新应用，新模型上线，都要重复一遍从零验证，成本快速累积。

所以需要一个能够把意图、实现、验证、逻辑链条串在一起的中间媒介，作为供需双方对齐的共同工作面。这个工作面不是大而全的超长上下文窗口，而是聚焦关键决策的放大镜，这就是 Elfiee 想要补上的空位。

# 3. 原因（Why）

1. 为什么现在做？

- 现在AI把实现一个需求的等待时间压缩到了极低，但是验证的成本已逐渐失控。
- elfiee的重点在于把意图、实现、验证和逻辑链条放在同一协作平台，遵循local-first的原则，让不同参与方可以在各自控制的数据域中，使用相同的协作和验证语言，可以应对数据主权带来的压力。
- AI时代的供应商涵盖数据，模型，基建（算力），场景（标准流程）等，相比传统的one2one交易模式，多方交易更普遍，需要一个平台，以清晰的需求和标准，促成更多交易。

2. 为什么现在可以做？

模型蒸馏、MOE等技术，可以使得场景、算法、数据等方，以最小公约数的基础设施条件，连接意图、验证和逻辑链条。且模型能力越强，成本越低，这种协同会越广泛和顺畅。

# 4. 需求（How）

可以从几个关键能力来描述 Elfiee 的设计需求：

1. 为多角色协作提供统一的工作面。（共同编辑的literature programming）

2. 所有角色的所有尝试都被记录且可回放。（event-sourcing，以及基于event的搜索，筛选，分类，回放）

3. 支持非线性的知识关系，还原逻辑链条。（DAG style relation，基于算法和向量时钟，对于event的因果关系（包括修改的event和引用的event），进行推断，聚类，搜索等）

4. 业务逻辑、检验标准和协作权限动态耦合（CBAC，可动态调整的权限模型，这是将.elf进行分享的核心内容，变成可执行的协作规则。在供需双方在同一个平面工作的前提下，自动执行从意图到实现到验证的流程）。

5. 分布式冲突检索。（event based vector clock，这是elf进行大规模推广的基础，可以使得更多的需求方/开发者，在离线协作的前提下，仍然可以精准定位冲突，进行修改）
